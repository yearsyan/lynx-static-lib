# lynxlib

`lynxlib` is now a CMake-driven wrapper around the official Lynx source tree.
The official code is included as a git submodule at `third_party/lynx`; the
top-level CMake targets synchronize pinned Lynx dependencies and then invoke
the upstream Windows GN/Ninja build.

## Layout

- `third_party/lynx`: official Lynx submodule pinned to `48546c5d0515ead65328d5f137a1d0860c06985f`
- `scripts/Sync-LynxDeps.ps1`: downloads pinned Habitat and runs Lynx dependency sync
- `scripts/Build-LynxWindows.ps1`: runs GN/Ninja and optionally creates `lynx_static.lib`
- `demo/src/lynx_static_demo.cc`: Win32 demo that links the generated static library
- `demo/bundle`: local Lynx demo page source built with the official rspeedy toolchain
- `third_party/official_deps.manifest.json`: reproducibility manifest
- `out/`: ignored official build output

## Bootstrap

```powershell
git submodule update --init --recursive
cmake --preset windows
cmake --build --preset deps
```

The dependency sync uses Habitat `0.3.147` from the official release URL and
checks SHA256 before executing it.

## Windows Toolchain

The Lynx Windows GN toolchain is clang-cl based. Although the upstream
`BUILDCONFIG.gn` has a branch for `is_clang=false`, this Lynx checkout only
instantiates `build/toolchain/win:clang_*` toolchains, so MSVC `cl.exe` is not a
drop-in replacement.

Install the Visual Studio component below before building on Windows:

```text
Microsoft.VisualStudio.ComponentGroup.NativeDesktop.Llvm.Clang
```

On this machine the installed toolchain is Visual Studio LLVM/Clang
`clang version 19.1.1`, component version `17.13.35710.127`. The upstream
`dependencies/DEPS` entry `buildtools/llvm` is pinned to `llvm-020d2fb7`, but
that official download is enabled only for Linux and Darwin; Windows uses
`VC\Tools\Llvm\x64\bin\clang-cl.exe` from Visual Studio.

## Build Targets

Build the official Windows SDK package:

```powershell
cmake --build --preset sdk
```

Build official Lynx object files and archive them into a static library. The
archive includes the official BoringSSL assembly helper library generated by GN:

```powershell
cmake --build --preset static
```

Expected static archive:

```text
out/lynx/Default/lynx_static.lib
```

Build the CMake Win32 demo. This creates a complete native window, creates a
LynxView through the public C API, loads a sample `.lynx.bundle`, and links
`out/lynx/Default/lynx_static.lib` as a normal COFF archive. The only forced
root symbol is the weak N-API host initializer required by PrimJS, so the linker
can still discard unused archive members:

```powershell
cmake --build --preset demo
```

Expected demo executable:

```text
build/cmake-driver/Release/lynx_static_demo.exe
```

The default demo bundle is built from source in this repository:

```text
demo/bundle/src/index.tsx
```

The build uses the official Node.js and rspeedy packages synchronized under the
Lynx submodule. `scripts/Build-DemoBundle.ps1` creates local Windows junctions
from `demo/bundle/node_modules` to the pinned packages in
`third_party/lynx/node_modules`, so the bundle build does not download or
resolve additional packages. The generated bundle is ignored build output under:

```text
demo/bundle/dist/main.lynx.bundle
build/cmake-driver/Release/resources/demo/main.lynx.bundle
```

Build the official upstream Win32 Explorer demo separately:

```powershell
cmake --build --preset explorer
```

Expected official Explorer executable:

```text
out/lynx/Default/lynx_explorer/lynx_explorer.exe
```

Build all of the above:

```powershell
cmake --build --preset all
```

## Git Hygiene

Commit the wrapper files, `.gitmodules`, and the submodule pointer. Do not
commit `out/`, `build/`, downloaded Habitat cache, or any generated Lynx
dependencies. The official Lynx submodule has its own ignore rules for synced
third-party dependencies.
